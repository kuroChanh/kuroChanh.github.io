Camera

This is an FPS (First Person Shooter) camera. The user can explore the game world in first person using the WASD keys. The the mouse wheel to zoom in or zoom out in the scene. This camera is updated when there is a change to the camera, recalculating the lookat matrix to render the scene properly to the user. If the user were to move the camera backwards, then the position of the camera would have to along the positive z-axis as that axis points towards the user. To go forward, the camera would move along the negative z-axis.

Multiple Lights

There is directional lighting that illuminates the game world. Point lights are used as the bridge lights as it lights up at one point. Spot light is used as a flash light for the user to use currently in this version. Each individual lighting in the scene is done using one shader, calculating the final output vector with the use of GLSL functions for each type of lighting. Once each light is calculated, they combine in order to create the final output vector, creating light in the scene.

Models

I use Assimp to import my .obj models into my scene. My mesh and model class are used to load and store the imported models. There's also some optimization where the functions can check whether the model has not been loaded. We check the file path and compare its path. If there is a match, then it will skip the rendering and use the located texture for the mesh.

Depth Test

I have a shader that is being used on some objects to show the use of the depth buffer. This depth buffer stores the information of each fragment and updates with each render. As the user moves closer to the object, it will become white. Moving away from the object will make it become black. This helps the user tell the depth between each object within the scene.

Geometry Shader

The geometry shader in this example makes an exploding affect on the model. To create this effect, the geometry calculates the normals of each face of the model, then uses the sin wave to change the distance of where the face is originally positioned, making the effect loop.

AABB Collision

Game objects can be given an AABB collision box. I created a class where AABB related information can be held and easily be added to a game object or anything else I needed to test this. By passing two game objects with AABB collision boxes to a function, they will be tested whether they have collided or not. If there is a collision, it will print a statement based on the results.

Dialogue Mechanic

I used Fire Emblem: Awakening as a reference and I was able to create my own dialogue mechanic. To read through the dialogue, the user presses enter which moves onto the next line. This mechanic uses a struct that has several vectors. These vectors include the dialogue lines, text background, characters 1 and 2, and bool. If the player finishes the dialogue, then it will stay on that final line. It will be unable to pass the size of the vector, which would cause an error if it did. This can be further optimized by creating an external file for the engine to read and input the values using a function, but I am happy with the current version!
